# Novel Proofreader

日本語の小説・ライトノベル原稿を対象とした、版組調整と校正・推敲を行うスキルです。

## このスキルの役割

商業出版の慣例に沿った版組への自動調整と、文章面の問題点の指摘・改善案提示を行います。創作物のため、自動で書き換えるのは形式面（版組）に限定し、内容面（校正・推敲）は必ず作者の判断を経るための指摘形式に留めます。

## 実行モード

デフォルトでは「版組修正 + 校正指摘」の両方を実行します。ユーザーの指示に応じて以下のように切り替えます：

- **両方**（デフォルト）: 版組修正ファイルと校正指摘ファイルの両方を生成
- **版組のみ**: ユーザーが「版組だけ」「typesetting only」等を指示した場合
- **校正のみ**: ユーザーが「校正だけ」「proofreading only」等を指示した場合

## 入出力

- **入力**: 原稿ファイル（テキスト形式、Markdown等。ファイル単位での処理）
- **版組修正の出力**: 元ファイルと同じディレクトリに `<元ファイル名>.typeset.<拡張子>` として別ファイル出力（元ファイルは保持）。ユーザーが出力先を指定した場合はそちらに従う
- **校正指摘の出力**: ユーザーが指定したパス（指定がなければ確認する）

## 作業開始時の確認

スキル起動時、以下を確認します：

1. **対象ファイル**: 処理する原稿ファイルのパス
2. **実行モード**: 両方／版組のみ／校正のみ
3. **校正指摘の出力先**: 両方または校正のみの場合
4. **作品固有ルールの有無**: キャラ名表記、用語集、文体方針などをまとめたファイル、または直接の指示。なければ「特になし」で進める

ユーザーのメッセージから1〜4が全て推定できる場合は、確認を省略して処理に入ってよい。

## 版組修正（自動修正）

商業出版の慣例に沿って **機械的に** 修正します。詳細なルールと検出パターンは `references/typesetting-rules.md` を参照してください。

> **重要**: 版組修正は形式面の機械的修正であり、「作者の意図」を理由にルールをスキップしてはならない。版組ルールに定義された修正項目は **検出→修正→検証** の手順で必ず実行する。作者の意図による例外は、ユーザーが明示的に指示した場合のみ認める。

### 修正項目と実行手順

版組修正は以下の **4カテゴリ** を、カテゴリごとに **検出→修正→件数記録** の順で実行する。「該当なし」と判断する前に必ず検出を行うこと。

#### 1. 記号類の正規化

**検出方法** (Grep ツールで各パターンを検索):
- 三点リーダー: `\.\.\.` (ASCII 3連), `…(?!…)` (単独の …)
- ダッシュ: `–` (en dash), `―` (horizontal bar), `--` (hyphen 2連)
- 感嘆符・疑問符: `[！？!?][^　」』\n]` (直後に全角スペース・閉じ括弧・改行がない箇所)

**修正内容**:
- 三点リーダー: `…` 単独や `...` → `……`（2連）に統一
- ダッシュ: En dash `–` や hyphen `--` → Em dash `——` に変換
- Em dash の連数: 行頭 → `——`（2連）必須、文中 → `—`（単独）も許容
- 感嘆符・疑問符: 直後に文が続く場合、全角スペース1つを挿入（文末・閉じ括弧前は不要）

#### 2. 段落・空行の整理

**検出方法**:
- 対象ファイルの総行数と、空行を除いたテキスト行数を比較する。差が大きい場合は Web 小説形式の可能性が高い
- 簡易判定: ファイル冒頭の10段落を Read で確認し、**全ての段落間に空行が1行ずつ入っているか** を目視で確認する

**修正内容**:
- Web小説形式の「全段落間に1行空け」を解消し、通常の改行のみの段落区切りに変換する
- **保持する空行**: 2行以上連続する空行、`---` や `◇` `◆` `＊` `* * *` などの区切り記号を含む行の前後（シーン転換の意図とみなす）
- 行頭の字下げ: 段落冒頭は全角スペース1字下げを基本（鉤括弧 `「` `『` やダッシュ `——` で始まる段落は字下げしない）

#### 3. 句読点の調整

**検出方法**:
- 代表的な地の文の段落を5つ選び、各文の読点の数を数える
- プロジェクトの CLAUDE.md に読点ルールが定義されている場合はそれを参照する

**修正内容**:
- 文節ごとに過剰に挿入された読点を間引く（1文あたり1〜3個を目安。CLAUDE.md に独自ルールがあればそちらに従う）
- **会話文（鉤括弧内）は原則変更しない**（キャラクターの個性表現）
- 詳細な削除/保持パターンは `references/typesetting-rules.md` セクション3を参照

#### 4. 長文の分割

**検出方法**:
- 100文字を超える文を目視で確認する（句点 `。` から次の句点までの文字数）

**修正内容**:
- 100文字超かつ複数の節を含む文を、自然な箇所（接続助詞、テ形接続）で分割する
- 著しく長い文のみが対象。文体として長文を選んでいる可能性も考慮し、保守的に適用する

### 版組修正の出力フォーマット

修正後の原稿を出力ファイルに保存し、ユーザーに以下を報告する：

```
版組修正完了: <出力ファイルパス>

修正内訳:
- 三点リーダー正規化: N箇所
- ダッシュ正規化: N箇所
- 感嘆符・疑問符後スペース: N箇所
- Web形式空行削除: N箇所
- 行頭字下げ調整: N箇所
- 過剰読点の調整: N箇所
- 長文分割: N箇所

保持された区切り（シーン転換等）: N箇所
```

## 校正・推敲（指摘＋改善案）

文章面の問題は自動修正せず、指摘と改善案の提示に留めます。詳細な観点とパターンは `references/proofreading-checks.md` を参照してください。

### 指摘の観点

**主要な指摘対象（積極的に拾う）**:
- 冗長な表現
- 説明不足（読者が状況を把握できない箇所）
- 主語と述語の関係が不明瞭
- 漢字・単語の誤用の可能性
- メタ表現の混入（ファイル名、「第1話」「プロローグ.md」など作品管理上の概念が本文中で言及されている）
- 表記揺れ（キャラクター名、地名、用語、固有名詞）
- 視点（POV）のブレ
- 時制の不自然な混在
- キャラクターの口調・一人称の揺れ
- 修飾語と被修飾語の距離が離れすぎている
- 「の」の連鎖（3つ以上連続）
- 受動態の濫用、二重否定

**控えめに指摘（よほど気になる時のみ）**:
- 文章のリズム（同じ語尾の連続など）
- 構造的問題（段落構成、シーン構成など）

**最低限に留める**:
- 指示語（あれ・これ・それ）の指摘 — ファイル単位の処理ではコンテキストが限定的なため

### 校正指摘の出力フォーマット

ユーザー指定パスにMarkdown形式で以下のように出力します：

```markdown
# 校正指摘: <原稿ファイル名>

実行日時: YYYY-MM-DD HH:MM
対象: <ファイルパス>

---

## 1. [カテゴリ名]（例: 冗長な表現）

### 1-1. <該当箇所の位置情報（行番号や見出し）>

**該当箇所**:
> 引用文

**指摘**: どこがどう問題か、簡潔に。

**改善案**:
> 改善後の文章

---

## 2. [次のカテゴリ]
...
```

カテゴリごとにまとめ、指摘の重要度や量に応じて並び順を調整します（重要なもの・修正コストが低いものを上位に）。

## 作品固有ルールの扱い

ユーザーから提供された作品固有のルール（キャラ名一覧、用語集、文体方針など）は、校正指摘の参照基準として利用します：

- キャラクター名・地名・用語の表記揺れ検出の基準
- 文体方針（語尾、一人称、口調など）の遵守チェック
- 指定された禁則表現の検出
- プロジェクトの `expression-rules.md` など、作品固有の表現ルールファイルがあれば読み込んで適用する

ルールが提供されない場合は、本文内での自己整合性のみをチェックします（例: 同じ名前が「アリス」と「ありす」で混在していないか）。

## 処理の流れ

### ステップ 1: 確認
対象ファイル、モード、出力先、作品ルールの有無を確認する。ユーザーのメッセージから全て推定できる場合はスキップ可。

### ステップ 2: 読み込み
- 原稿ファイルを読み込む
- 作品固有ルールがある場合はそれも読み込む
- `references/typesetting-rules.md` と `references/proofreading-checks.md` を読み込む

### ステップ 3: 版組修正の事前検出（モードに応じて）

**このステップは省略不可。** 修正対象の4カテゴリそれぞれについて、実際にファイル内を調べて問題箇所の有無と規模を把握する。

各カテゴリの検出方法は「版組修正 > 修正項目と実行手順」に記載。

検出結果の例:
```
事前検出結果:
- 記号類: 三点リーダー異常 0件, ダッシュ異常 0件, 感嘆符後スペース 0件
- 段落・空行: Web形式空行あり (推定 200行), 字下げ異常 0件
- 句読点: 地の文の読点密度 平均5.2個/文 (目安の1〜3個を超過)
- 長文: 100文字超の文 2件
```

### ステップ 4: 版組修正の実行（モードに応じて）

事前検出の結果に基づき、検出された問題を修正する。修正は以下の順序で適用する:

1. 区切り記号・複数行空行の保護（変更されないようマーク）
2. 三点リーダーの正規化
3. ダッシュの正規化と連数調整
4. Web形式の段落間1行空けを削除
5. 行頭字下げの調整
6. 感嘆符・疑問符後のスペース挿入
7. 過剰な読点の間引き
8. 長文の分割

修正後のファイルを出力先に書き出す。

### ステップ 5: 版組修正の出力検証（省略不可）

出力ファイルの **冒頭20行程度を Read で読み**、以下を確認する:

- [ ] 段落間の空行が削除されているか（Web形式空行があった場合）
- [ ] `---` 前後の空行が保持されているか
- [ ] 読点の間引きが反映されているか（修正前後を比較）
- [ ] タイトル行や字下げが崩れていないか

検証に失敗した場合は、該当箇所を修正して再出力する。

### ステップ 6: 修正サマリの報告

修正内訳を件数付きでユーザーに報告する。フォーマットは「版組修正の出力フォーマット」に従う。

### ステップ 7: 校正指摘の実行（モードに応じて）

`references/proofreading-checks.md` を参照しながら問題箇所を抽出し、カテゴリ別に整理して指摘ファイルに出力する。主要な指摘の概要を会話内でも報告する。

---

## 複数ファイルの処理

複数ファイルを処理する場合は、Agent ツールで並列処理してよい。各エージェントには以下を渡す:

- 担当ファイルのパス
- 版組修正ルールの要約（読点間引きルール、空行削除ルール等）
- 出力先パス
- 会話文は変更しない旨の注意

エージェントの完了後、出力ファイルの検証（ステップ5）を必ず行う。

## 注意事項

### 版組修正について
- 版組修正は **機械的な形式修正** であり、「作者の意図」「文体の特徴」を理由にルールの適用をスキップしない
- 「該当なし」と判断する前に、必ず Grep やファイル読み込みによる検出を行う
- 検出の結果、本当に該当箇所がない場合のみ「0件」と報告する
- 会話文（鉤括弧内）は版組修正の対象外（読点間引き等を適用しない）

### 校正・推敲について
- **創作物の尊重**: 文体、リズム、独特な表現は作者の意図である可能性が高いため、明らかな誤りや一貫性の欠如以外は指摘しない
- **過剰指摘の回避**: 全ての箇所を指摘するのではなく、改善効果の高いものに絞る
- **作中の世界観優先**: ファンタジー作品等で意図的に造語や古めかしい表現が使われている場合、それを誤用として扱わない
