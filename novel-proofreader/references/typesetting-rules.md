# 版組ルール詳細

日本語商業出版小説の慣例に沿った版組調整の詳細ルールです。

## 1. 記号類の正規化

### 1.1 三点リーダー

**ルール**: 三点リーダーは `……`（U+2026 を2つ連続）に統一する。

**検出・修正パターン**:

| 入力パターン | 修正後 |
|---|---|
| `...` (ASCII period × 3) | `……` |
| `．．．` (全角ピリオド × 3) | `……` |
| `・・・` (中黒 × 3) | `……` |
| `…` (単独) | `……` |
| `………` (3連以上、奇数) | `……` または `……………`（偶数になるよう調整） |

**注意**:
- 既に `……` になっている箇所は変更しない
- 偶数個の `…` 連続は保持（`……` `…………` など）
- 奇数個の場合は1つ追加して偶数化

### 1.2 ダッシュ

**ルール**: ダッシュには Em dash `—` (U+2014) を用いる。

**検出・修正パターン**:

| 入力パターン | 文脈 | 修正後 |
|---|---|---|
| `–` (En dash) | 全般 | `—` |
| `--` (hyphen × 2) | ダッシュ意図 | `——` |
| `―` (Horizontal bar) | 全般 | `—` に統一 |
| `ー` (長音符) | 文脈上ダッシュ意図 | 文脈判断、通常は変更しない（長音符の可能性が高いため）|

### 1.3 Em dash の連数

**ルール**:
- **行頭にダッシュがある場合**: `——`（2連）必須
- **文中のダッシュ**: `—`（単独）も `——`（2連）も許容、ただし作者が一貫している場合はその方針を尊重

**判定基準**:
- 行頭: 段落の最初の文字、または鉤括弧の直後
- 文中: それ以外

**修正例**:
```
修正前: —そして彼は去った。
修正後: ——そして彼は去った。

修正前（文中、単独で許容）: 彼は—おそらく—頷いた。
修正後: 変更不要（許容範囲）
```

### 1.4 感嘆符・疑問符の後の空白

**ルール**: `！` `？` の直後に文が続く場合、全角スペース1つを挿入する。

**例外**:
- 文末（次が句点や閉じ括弧、改行）の場合は不要
- 連続する記号（`！？` `！！`）の途中は不要、連続の最後の記号の後で判断

**例**:
```
修正前: 「待って！」と彼女は叫んだ。
修正後: 変更不要（閉じ括弧の前なので）

修正前: 「待って！どこへ行くの？」
修正後: 「待って！　どこへ行くの？」

修正前: 「本当！？信じられない！」
修正後: 「本当！？　信じられない！」
```

## 2. 段落と空行の整理

### 2.1 Web小説形式の段落間1行空け

**問題**: なろう系・カクヨム系の慣行で「全ての段落の間に空行を1行入れる」スタイルは、商業出版では一般的ではない。

**ルール**: 段落間の1行空けを削除し、通常の改行のみの段落区切りに変換する。

**判定**:
- 「1行空け」= 空行（改行のみの行）が1行だけ挟まれている状態
- 「複数行空け」= 空行が2行以上連続 → **保持**（シーン転換等の意図とみなす）

**例**:
```
修正前:
　彼は森を歩いていた。

　遠くで鳥が鳴いている。

　風が梢を揺らした。

修正後:
　彼は森を歩いていた。
　遠くで鳥が鳴いている。
　風が梢を揺らした。
```

### 2.2 保持する空行・区切り

**以下のパターンは作者の意図的な区切りとして保持する**:

1. **2行以上連続する空行**
2. **区切り記号を含む行**:
   - `---` `***` `___` のような Markdown 水平線
   - `◇` `◆` `◯` `●` `※` `＊` `*` の単独行または連続
   - `* * *` `◇ ◇ ◇` のようなスペース区切り並び
   - `〜〜〜` の連続

これらは前後の空行も含めて保持する。

### 2.3 行頭の字下げ

**ルール**: 段落冒頭は全角スペース1字（`　`）の字下げを基本とする。

**例外**:
- 鉤括弧 `「` `『` で始まる段落は字下げしない
- 既に字下げ済みの箇所は変更しない（重複させない）
- ダッシュ `——` で始まる段落は字下げしない

**判定**: 段落の最初の文字を確認
- 通常文字で始まり字下げなし → 字下げ追加
- `「` `『` `——` で始まる → 字下げ不要（あれば削除）

**注意**: 一行目から字下げを意図的に省く文体もあるため、原稿全体で既に字下げが行われていない場合は、字下げを追加しない（既存方針の尊重）。判断に迷う場合は、原稿の最初の数段落の状態を確認して方針を決める。

## 3. 句読点の調整

### 3.1 読点（、）を打つ条件

**原則**: 読点は以下の4ケースに該当するときのみ打つ。それ以外では打たない。文節の区切りごとに惰性で打たれた読点は削減対象とする。

1. **主題・主語の後で意味の区切りが必要なとき**
   例: 「彼は、ゆっくりと振り返った」
2. **接続助詞や逆接の後**
   例: 「行きたかったが、足が動かなかった」
3. **並列要素の区切り**
   例: 「赤と、青と、白の旗」
4. **読点がないと係り受けを誤読しうるとき**

上記のいずれにも当てはまらない読点は、原則として削除する。

### 3.2 過剰読点（NG例）と修正

文節ごとに読点を撒いた文はリズムが細切れになる。明確なNG例:

```
×「両手で、受け取って、自分の中の、何か、薄く、温かいところに、ゆっくり、しまった。」
○「両手で受け取って、自分の中の薄く温かいところにゆっくりとしまった。」
```

### 3.3 判断基準（1文3読点ルール）

**1文に読点が3つ以上必要だと感じたら、その文は構造に無理がある。** 次のいずれかで対処する:

- 文を2文以上に分割する
- 係り受け（修飾関係）を見直して、読点なしで読める語順に組み替える

3つ以上の読点を残したまま「全部必要」と判断するのは原則として誤り。まず分割・組み替えを試みること。ただし 3.1-3 の並列列挙のみで構成された文（A、B、C、D…）はこの限りでない。

### 3.4 適用上の注意

- **会話文（鉤括弧内）は原則変更しない**（キャラクターの個性・話し方の表現のため）
- 削減・組み替えで文意が変わらないことを各箇所で確認する。文意が変わる場合は語順の組み替え（3.3）を優先し、安易に読点だけ消さない

## 4. 長文の分割

### 4.1 判定基準

**対象**: 以下を全て満たす文
- 100文字以上
- 複数の節（主述関係）を含む
- 自然な分割点が存在する

### 4.2 分割の方針

- **分割点**: 接続助詞（〜が、〜ので、〜けれど）、並列の読点、テ形接続の後
- **分割しない**: 短い修飾節の途中、引用の途中、固有表現の途中
- **保守的に**: 著しく長い文のみを対象とし、文体として長文を選んでいる可能性も考慮

**例**:
```
修正前: 彼は森の中を歩きながら、空を見上げて鳥の声を聞いていたが、ふと足元に視線を落とすと、そこには見たこともない不思議な花が咲いていて、その花弁は朝露を受けて静かに光っていた。

修正後: 彼は森の中を歩きながら、空を見上げて鳥の声を聞いていた。だがふと足元に視線を落とすと、そこには見たこともない不思議な花が咲いていた。その花弁は朝露を受けて静かに光っていた。
```

**注意**: 長文分割は版組修正の中で最も判断を要するため、明白なケースのみに適用する。迷う場合は校正側の指摘（リズムの観点）として扱う。

---

## 5. 修正の優先順位と注意

### 修正の適用順序

1. 区切り記号・複数行空行の保護（変更されないようマーク）
2. 三点リーダーの正規化
3. ダッシュの正規化と連数調整
4. Web形式の段落間1行空けを削除
5. 行頭字下げの調整
6. 感嘆符・疑問符後のスペース挿入
7. 過剰な読点の間引き
8. 長文の分割

### 修正サマリの形式

修正後、以下のような形式で報告する：

```
版組修正完了: <出力ファイルパス>

修正内訳:
- 三点リーダー正規化: N箇所
- ダッシュ正規化: N箇所
- Em dash 行頭2連化: N箇所
- Web形式空行削除: N箇所
- 行頭字下げ追加: N箇所
- 感嘆符・疑問符後スペース: N箇所
- 過剰読点の調整: N箇所
- 長文分割: N箇所

保持された区切り（シーン転換等）: N箇所
```

### 修正しない方が良いケース

- 原稿全体で一貫した独自スタイルが採用されている場合（例: 全文字下げなし、独自の記号体系）
- ジャンルとして特殊な慣行がある場合（実験的文学、詩的散文など）
- ユーザーから明示的に「このスタイルは保持して」と指示された項目

判断に迷う場合は、修正前にユーザーに確認する。
